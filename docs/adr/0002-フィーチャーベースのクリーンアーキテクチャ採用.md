# ADR-0002: フィーチャーベースのクリーンアーキテクチャ採用

| 項目       | 内容       |
| ---------- | ---------- |
| ステータス | Accepted   |
| 日付       | 2026-03-31 |

---

## コンテキスト

Go/Ginで構築するstock_backendのディレクトリ構成と責務の分離方針を決定する必要があった。

このAPIは複数のフィーチャー（認証、ローソク足データ、銘柄リスト、ウォッチリスト、ロゴ検出）を持ち、それぞれが独自のドメインロジック・データアクセス・HTTPハンドリングを持つ。将来的なフィーチャー追加や既存フィーチャーの独立したメンテナンスを見据えた設計が必要であった。

また、テスタビリティを確保するため、ビジネスロジックをHTTP層やデータベース層から切り離すことが求められた。

## 決定

**フィーチャーベースの垂直スライス（Vertical Slices）** を基本構造として採用し、各フィーチャー内部では **クリーンアーキテクチャのレイヤー分離**（domain → usecase → adapters → transport）を適用する。

依存関係は以下の方向のみ許可する：

```
transport → usecase → domain
adapters  → domain（usecaseのインターフェース実装）
```

リポジトリインターフェースは **usecase層（利用者側）で定義する**（Goの「インターフェースは利用者が定義する」慣例に従う）。

## 理由

**フィーチャーベース構成によるコンテキストの局所性**
`auth`、`candles`、`symbollist` などの機能を横断的なレイヤー（`handlers/`、`repositories/`）ではなく、フィーチャー単位（`feature/candles/`）でまとめることで、特定機能に関するコードが一箇所に集約される。新機能追加や既存機能の修正時に参照・変更すべきファイルが明確になる。

**ドメインロジックの独立性**
domain層にはGoの純粋な構造体のみを配置し、フレームワーク・ORMへの依存を持たせない。これによりビジネスロジックのユニットテストが外部依存なしに実行できる。

**インターフェースによるテスタビリティ**
usecase層がリポジトリインターフェースを定義し、adapters層が実装することで、テスト時にモックリポジトリを注入できる。HTTPハンドラーのテストもusecaseインターフェースをモックすることで、HTTPレイヤーの単体テストが可能になる。

**Goの慣例に沿ったインターフェース定義**
Goでは「インターフェースは利用者が定義する」ことが推奨される。usecase層が必要とする最小限のインターフェース（例: 読み取り専用の`CandleRepository`と書き込み専用の`CandleWriteRepository`）を定義することで、インターフェース分離の原則を自然に実現できる。

**golangci-lintによる依存ルールの自動強制**
`depguard`を使用し、domain/usecaseからadapters/transportへの不正インポートをCIで自動検出する。アーキテクチャルールがコードレビューに依存せず機械的に守られる。

**フィーチャー間の疎結合**
各フィーチャーが他のフィーチャーをインポートしないルールを`depguard`で強制することで、フィーチャー間の意図しない依存が生まれることを防ぐ。共有が必要なものは`internal/shared/`または`internal/platform/`に配置する。

## 代替案

| 代替案                                         | 不採用の理由                                                                                                                                 |
| ---------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| レイヤーベース構成（`handlers/`・`repositories/`・`models/` をトップレベルに配置） | 1つの機能に関するコードが複数のトップレベルディレクトリに散在し、機能の追加・削除時に複数箇所の変更が必要になる。フィーチャーが増えると見通しが悪化する |
| Hexagonal Architecture（ポートとアダプター）の厳密な適用 | フィーチャーベース垂直スライスと概念的に近いが、「ポート」「アダプター」という用語とディレクトリ構成がGoのエコシステムでは定着していない。本プロジェクトの規模では過剰になる可能性がある |
| DDD（ドメイン駆動設計）の集約・バウンデッドコンテキスト厳密適用 | ビジネスドメインが明確に分離されているこのプロジェクトには適合する部分もあるが、集約・ファクトリ等の追加概念の学習コストと、Goの値型中心設計との相性を考慮してよりシンプルなクリーンアーキテクチャ適用にとどめる |
| フラット構成（パッケージ分割なし）             | 初期は単純だが、コードが増えるにつれて責務の境界が不明確になり、テスタビリティが低下する                                                   |

## 影響

### ポジティブな影響

- 新フィーチャーの追加手順が明確になる（CLAUDE.mdに「新機能の追加」として文書化済み）
- usecase層のユニットテストが外部依存なしに実行できる
- HTTPハンドラーのテストがモックusecaseで独立して実行できる
- `depguard`により、アーキテクチャ違反がCIで自動検出される
- フィーチャー単位でREADME.mdを配置することで、機能ごとのドキュメントが局所化される
- デコレータパターン（`CachingCandleRepository`）がusecaseコードを変更せずに透過的に適用できる

### ネガティブな影響・トレードオフ

- 新フィーチャー追加時に `.golangci.yml` の `depguard` 設定を手動で更新する必要があり、忘れると依存ルールが機能しなくなる
- フィーチャー間で類似したリポジトリパターンのコードが重複しがちで、共通化の判断が必要になる場面がある
- 依存性注入（DI）を `cmd/server/main.go` で手動で行うため、フィーチャーが増えるにつれてmain.goが長くなる傾向がある
- `internal/api/types.gen.go`（OpenAPIから自動生成）がtransport層のみで使用可能というルールを徹底する必要があり、新規開発者が誤ってusecase層で使用するリスクがある
