# ADR-0006: DB 操作を GORM から sqlc + goose に移行

| 項目       | 内容       |
| ---------- | ---------- |
| ステータス | Accepted   |
| 日付       | 2026-05-24 |

---

## コンテキスト

これまで本プロジェクトは GORM（`gorm.io/gorm`）を ORM として採用しており、
DB アクセスとマイグレーション（`AutoMigrate` + 各 feature の `migration.go` で FK 追加）の
両方を GORM 経由で行っていた。

運用と保守を進める中で以下の課題が顕在化した。

1. **マイグレーションが構造体依存**: `AutoMigrate` は構造体のフィールド追加・型変更を
   推測するため、本番運用で危険な ALTER 文を発行しないか目視レビューが必要だった。
   さらに `NOT NULL` 解除や FK 追加・複雑な変更には対応できず、別途 `migration.go` を
   feature ごとに書いて補完していた。
2. **DDL の履歴が散在**: スキーマの正本が「構造体タグ ＋ feature ごとの migration.go ＋
   実 DB」に分散しており、現在のスキーマや差分が一目で追えなかった。
3. **クエリの型安全性が弱い**: GORM のチェイン API は実行時まで誤りを検出しにくく、
   カラム名や JOIN 条件のタイポがプロダクションでエラーになる。
4. **テストが SQLite 依存**: リポジトリテストの一部が GORM の SQLite driver を使っており、
   方言差（PostgreSQL 固有の機能・制約挙動・エラーコード）を再現できなかった。

## 決定

以下の構成に移行した。

- **マイグレーション**: [pressly/goose](https://github.com/pressly/goose) を採用し、
  `db/migrations/*.sql` を SQL ファイルベースで管理する。
- **DB アクセス**: [sqlc](https://github.com/sqlc-dev/sqlc) を採用し、SQL ファイルから
  型安全な Go コードを生成する。`database/sql` + `github.com/jackc/pgx/v5/stdlib` driver
  をランタイムで使用する。
- **テスト**: `github.com/testcontainers/testcontainers-go` で実 PostgreSQL コンテナを
  立ち上げ、ローカル/CI 双方で本番と同じ DB を使う。`TEST_DB_DSN` 環境変数で既存の
  PostgreSQL を流用することも可能。
- **GORM は完全に除去**し、`go.mod` および depguard で再混入を禁止する。
- **`cmd/server` 起動時マイグレーションは廃止**し、専用バイナリ `cmd/migrate`
  （Dockerfile.migrate でビルド）を Cloud Run Job 等で別途実行する運用に変更する。

## 理由

| 観点                 | 移行先構成の優位性                                                                 |
| -------------------- | ---------------------------------------------------------------------------------- |
| マイグレーション安全 | DDL を直接 SQL で書くため、何が起きるか即座に分かる。レビューしやすい              |
| スキーマの正本       | `db/migrations/00001_init_schema.sql` を起点とした単一の SQL 履歴                  |
| 型安全               | sqlc がカラム名・型・引数を Go コードに反映するため、ビルド時にズレを検出できる    |
| 方言の正確性         | テストも本番も PostgreSQL なので、ON CONFLICT・FK・エラーコード等の挙動が一致      |
| 学習コスト           | プレーン SQL + 生成された関数を呼ぶだけで、ORM の独自 API を覚えなくて済む         |
| デプロイ統制         | マイグレーションを独立ジョブにし、サーバー起動と切り離すことでロールバックが容易   |

## 代替案

| 代替案                          | 不採用の理由                                                                                   |
| ------------------------------- | ---------------------------------------------------------------------------------------------- |
| GORM の継続利用 + 手動 SQL 補強 | スキーマの正本問題と型安全性の弱さが解消しない                                                 |
| ent (Facebook 製 ORM)            | コード生成型で型安全だが、独自スキーマ DSL を覚える必要があり、移行コストが本案より高い        |
| sqlx + golang-migrate           | 候補だが、sqlc のような完全な型安全コード生成は無く、SQL ↔ Go の重複が残る                     |
| Bun ORM                          | 比較的軽量だが GORM と同様の課題（実行時エラー、AutoMigrate 由来のスキーマ管理）が残る         |

## 影響

### ポジティブな影響

- スキーマ変更履歴が `db/migrations/*.sql` に一元化され、差分レビューが容易になった。
- リポジトリ実装が SQL に近くなり、実行計画やインデックス設計を把握しやすくなった。
- テストが本物の PostgreSQL を使うため、本番と同じエラーコード・制約挙動を検証できる。
- マイグレーションをデプロイから切り離したことで、適用失敗時の影響範囲が `cmd/migrate` の
  ジョブに閉じる。

### ネガティブな影響・トレードオフ

- テスト起動に Docker（testcontainers-go）が必要になり、ローカル/CI のセットアップ要件が
  増えた。`TEST_DB_DSN` で既存 PostgreSQL を使う逃げ道は用意してある。
- 各 feature の `adapters/sqlc/` 配下に生成コードが増え、`go tool sqlc generate` の
  実行を忘れると CI が落ちる運用負荷が増える。
- マイグレーション適用が独立したジョブになったため、デプロイパイプラインの調整が必要に
  なった（Cloud Run Job の前段実行を Workflow に組み込む等）。
