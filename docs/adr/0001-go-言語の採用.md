# ADR-0001: Go言語の採用

| 項目       | 内容       |
| ---------- | ---------- |
| ステータス | Accepted   |
| 日付       | 2026-03-31 |

---

## コンテキスト

株式データ配信・ユーザー認証を提供するバックエンドAPIを新規に構築する必要があった。このAPIは以下の特性を持つ：

- 外部マーケットデータAPI（TwelveData）への定期的なバッチ取り込み
- JWTによる認証・認可ミドルウェア
- RedisキャッシュとPostgreSQLを組み合わせたデータ層
- フロントエンド（Kotlin / Jetpack Compose）からのリクエストを受け付けるREST API
- Docker上での実行、Cloud Runへのコンテナデプロイを前提

言語の選定にあたり、開発効率・実行効率・長期保守性を重視した評価を行った。

## 決定

バックエンドAPIの実装言語として **Go（golang）** を採用する。
Webフレームワークには **Gin** を使用する。

## 理由

**静的型付けと高いパフォーマンス**
Goはコンパイル型の静的型付け言語であり、実行時のパフォーマンスが高い。株価データの一括取り込みや多数の並列リクエスト処理において、GCによる一時停止が少なくメモリ効率が良い。

**並行処理の簡潔な表現**
goroutineとchannelにより、外部APIへのレート制限付きバッチ処理や複数のリクエスト並列処理を、スレッド管理の複雑さなしに実装できる。本プロジェクトのレートリミッター付きデータ取り込みはこの恩恵を直接受ける。

**単一バイナリによるシンプルなデプロイ**
`go build` で生成される単一の静的リンクバイナリにより、Dockerイメージを極めて小さく保てる（distrolessベースイメージが使用可能）。コンテナ環境（Cloud Run）へのデプロイが単純化される。

**標準ライブラリの充実**
`net/http`、`log/slog`（構造化ログ）、`context`による伝播、`crypto`などがすべて標準ライブラリに含まれており、外部依存を最小限に抑えられる。

**Ginフレームワークの適合性**
Ginは高パフォーマンスなHTTPルーターであり、ミドルウェアチェーン・パラメータバインディング・JSONレスポンスヘルパーがAPIサーバー開発に適合している。

**エコシステムの成熟**
GORM（PostgreSQL）、go-redis、golang-jwt、oapi-codegen（OpenAPIコード生成）、golangci-lintなど、このプロジェクトで必要なすべての領域で成熟したライブラリが揃っている。

## 代替案

| 代替案                       | 不採用の理由                                                                                             |
| ---------------------------- | -------------------------------------------------------------------------------------------------------- |
| Kotlin (Ktor / Spring Boot)  | フロントエンドがKotlinのため統一できる利点はあるが、JVMの起動時間・メモリ消費がコンテナ環境で不利。Goの方がコールドスタートが速い |
| Python (FastAPI / Django)    | 動的型付けによる実行時エラーリスク、GILによる並行処理の制約。スクリプト言語としての適性は高いが、型安全性と実行効率でGoが優位 |
| Node.js (Express / Fastify)  | 単一スレッドイベントループによるI/O処理は得意だが、型安全性にはTypeScriptが必要で複雑性が増す。Goの方がシンプルに並行処理を扱える |
| Rust                         | 最高レベルの実行効率と安全性を持つが、学習コスト・開発速度・Webフレームワークの成熟度の観点でAPIサーバー開発には過剰 |

## 影響

### ポジティブな影響

- コンパイル時の型チェックにより、実行時エラーを事前に排除できる
- 単一バイナリのDockerイメージで、デプロイパイプラインが簡潔になる
- goroutineにより、レート制限付きバッチ取り込みを可読性高く実装できる
- 標準ライブラリの `log/slog` で構造化ログが追加依存なしに実現できる
- golangci-lintによる静的解析でコード品質を自動的に担保できる

### ネガティブな影響・トレードオフ

- エラーハンドリングが明示的で冗長（`if err != nil` の繰り返し）になりがちで、コード量が増える
- ジェネリクス（Go 1.18+）は追加されたが、他言語に比べてまだ表現力に制約がある
- Goに不慣れな開発者が参加する場合、goroutineのリークやcontextの伝播ルールの習得が必要
- `null` の概念がなくポインタで表現するため、nilポインタデリファレンスへの注意が必要
