# ADR-0004: tbls による ER 図自動生成

| 項目       | 内容       |
| ---------- | ---------- |
| ステータス | Accepted   |
| 日付       | 2026-04-20 |

---

## コンテキスト

stock_backend のスキーマは GORM の `AutoMigrate` により、アプリケーション起動時にエンティティ（`internal/feature/**/domain/entity/`）と一部アダプタ内のモデル（例: `CandleModel`）から生成される。
SQL マイグレーションファイルのような「スキーマの単一ソース」が存在せず、以下の課題があった：

- DB 全体像を把握できるドキュメント（ER 図・テーブル定義書）が存在しない
- フィーチャー追加やカラム変更時に、全体のリレーション（例: `watchlists → users`, `watchlists → symbols`）を俯瞰しづらい
- `cmd/server/main.go` と `internal/feature/watchlist/adapters/migration.go` に散在する FK 制約の追加を、エンティティ定義と突き合わせて確認する手段がない

一方で、手書きの ER 図は更新が追いつかず陳腐化する懸念がある。スキーマ変更時に自動で最新化される仕組みが必要だった。

## 決定

スキーマドキュメント生成ツールとして **[tbls](https://github.com/k1LoW/tbls)** を採用する。

- 設定ファイル: [.tbls.yml](../../.tbls.yml)（リポジトリルート）
- 出力先: [docs/schema/](../schema/)（Git 管理、PR レビュー対象）
- ER 図形式: **Mermaid**（GitHub 上でネイティブレンダリング、差分レビュー可能）
- 生成元: 稼働中の PostgreSQL から `INFORMATION_SCHEMA` を読み取る（ランタイムの実スキーマが単一ソース）
- CI ジョブ `Schema Doc Drift` で `tbls diff` を実行し、スキーマと `docs/schema/` の乖離がある PR を失敗させる
- CI でマイグレーションだけを実行するための小さなバイナリ [cmd/migrate](../../cmd/migrate/) を新設する（`cmd/server` は GCP クライアント初期化を含むため CI では使えない）

## 理由

**GORM ベースのプロジェクトに適合する**
tbls は SQL マイグレーションファイルではなく実際の DB スキーマから読み取る方式のため、`AutoMigrate` でスキーマが生成されるこのプロジェクトと相性がよい。エンティティの GORM タグ（`uniqueIndex` など）を変更すれば、マイグレーション後に自動で反映される。

**Mermaid による GitHub ネイティブレンダリング**
Mermaid 形式の ER 図は GitHub 上でそのままレンダリングされ、Markdown diff としてレビュー可能。PlantUML/SVG と異なり graphviz などの追加依存が不要で、生成物サイズも小さい。

**CI ドリフト検出による陳腐化防止**
`tbls diff` を CI に組み込むことで、スキーマ変更時に `docs/schema/` の再生成を忘れた PR を機械的に検出できる。「ドキュメント更新は人間の規律に依存する」という常態的な陳腐化リスクを防げる。

**cmd/migrate の副次的な価値**
CI 用に新設した `cmd/migrate` は、Cloud Run のデプロイ前マイグレーションジョブとしても再利用できる。現状は `cmd/server` の起動時に `RUN_MIGRATIONS=true` で実行しているが、これは起動時間を伸ばし、かつ複数インスタンス起動時の競合リスクもあるため、将来的に分離する布石となる。

## 代替案

| 代替案                            | 不採用の理由                                                                                                       |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| SchemaSpy                         | Java 製で重い。HTML 出力が中心で GitHub での diff レビューに向かない                                               |
| dbdiagram.io（DBML）              | 手書きの DBML をメンテナンスする必要があり、スキーマとの乖離が起こりやすい                                          |
| Mermaid を手書きで ADR に記載     | スキーマ変更時に人手で更新する必要があり、陳腐化が避けられない                                                      |
| PlantUML 形式で tbls を使用       | SVG のため diff レビューが難しく、graphviz 依存も発生する。Mermaid で十分表現できる                                 |
| エンティティから Go コードで生成  | gorm.io/gen などで独自ツールを書くのはメンテナンスコスト高。tbls は実績があり十分な機能を持つ                      |

## 影響

### ポジティブな影響

- 開発者が `docs/schema/README.md` を見れば、DB 全体のテーブル構成・カラム・FK 制約・インデックスを即座に把握できる
- スキーマ変更を含む PR では `docs/schema/` の diff が同時に表示され、レビュアーが変更の影響範囲を視覚的に追える
- CI ドリフト検出により、ドキュメントの陳腐化が構造的に防がれる
- `cmd/migrate` の新設により、将来的に Cloud Run pre-deploy ジョブとしてマイグレーションを分離する選択肢が開ける

### ネガティブな影響・トレードオフ

- スキーマを変更した PR では `docs/schema/` の再生成とコミットが必須となり、変更一つあたりの手順が一段増える
- 生成物を Git 管理するため、大規模なスキーマ変更時はコミット差分が大きくなる（ただしレビュー可能性のトレードオフとして受容）
- tbls の実行には稼働中の PostgreSQL が必要なため、ローカル再生成には Docker Compose の起動が前提となる
