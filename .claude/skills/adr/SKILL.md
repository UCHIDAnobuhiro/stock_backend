アーキテクチャ上の決定を記録するADRを生成してください。

引数: $ARGUMENTS

## 手順

### Step 1: トピックの確認

1. 引数にトピックが指定されている場合はそれをADRのテーマとして使用する
2. 引数がない場合は、何について記録するかをユーザーに確認する
3. トピックが曖昧な場合は「どの技術・設計の決定を記録したいか」を1文で聞く

### Step 2: 既存ADRの確認と番号採番

1. `docs/adr/` ディレクトリ内のファイルを確認する（存在しない場合はディレクトリ作成が必要なことをメモする）
2. `NNNN-*.md` パターンに合致するファイルの最大番号を特定する（例: `0002-xxx.md` があれば次は `0003`）
3. ADRが1件もない場合は `0001` から開始する
4. 新しい番号を4桁ゼロ埋めで採番する（例: `0003`）

### Step 3: コードベースの調査（必要に応じて）

トピックに関連するコードやドキュメントを調査し、ADRに盛り込むべき情報を収集する：

- `CLAUDE.md` を参照して現在のアーキテクチャ方針を把握する
- 関連するフィーチャーディレクトリ（`internal/feature/<name>/`）を確認する
- `go.mod` から採用している依存ライブラリを確認する
- 既存のADR（`docs/adr/`）を参照して関連するものがないか確認する

### Step 4: ADR草案の生成

以下のテンプレート（`docs/adr/template.md`）に従い、収集した情報をもとにADRの草案を生成する。

**ファイル名の決定**:

- 形式: `NNNN-kebab-case-title.md`
- タイトル部分は日本語または英語のケバブケース
- 例: `0003-postgresql-の採用.md`、`0004-redis-cache-strategy.md`

**テンプレート構造**:

```markdown
# ADR-NNNN: タイトル

| 項目       | 内容       |
| ---------- | ---------- |
| ステータス | Proposed   |
| 日付       | YYYY-MM-DD |

---

## コンテキスト

（背景・問題・制約を記述）

## 決定

（何を決定したかを1〜3文で記述）

## 理由

（なぜこの選択をしたか。評価した観点ごとに説明）

## 代替案

| 代替案 | 不採用の理由 |
| ------ | ------------ |
| 〜     | 〜           |

## 影響

### ポジティブな影響

- 〜

### ネガティブな影響・トレードオフ

- 〜

## 関連ADR

- [ADR-XXXX](XXXX-title.md): 関連内容
```

**記述ルール**:

- すべて日本語で記述する
- ステータスは `Proposed`（草案）で開始する。ユーザーが `Accepted` を指定した場合はそちらを使用する
- 日付は今日の日付（YYYY-MM-DD形式）を使用する
- 「関連ADR」セクションは関連するものがなければ省略する
- コンテキスト・理由・影響は具体的に記述し、抽象的な記述を避ける

### Step 5: 草案の提示と確認

生成した草案をユーザーに表示し、以下を確認する：

```text
## ADR草案

ファイル名: docs/adr/NNNN-title.md

---
（ADRの内容全文）
---

保存してよいですか？修正が必要な箇所があれば教えてください。
```

ユーザーから修正の指示があれば内容を更新し、再度確認を取る。

### Step 6: ファイルの保存

ユーザーの承認後、以下を実行する：

1. `docs/adr/` ディレクトリが存在しない場合は作成する
2. 決定したファイル名でADRを保存する
3. `docs/adr/README.md` の一覧テーブルに新しいADRを追記する（`README.md` が存在する場合）
4. 保存完了をユーザーに通知する：

```text
ADRを保存しました: docs/adr/NNNN-title.md

docs/adr/README.md の一覧も更新しました。
次のステップとして /commit でコミットすることをお勧めします。
```

## 注意事項

- 保存前に必ずユーザーの確認を取る
- ステータスは基本的に `Proposed` で作成し、レビュー後に `Accepted` へ変更する運用を推奨する
- 既存のADRを変更する場合は、元のADRのステータスを `Superseded by ADR-NNNN` に更新する
- ADRは一度 `Accepted` になったら内容を書き換えず、新しいADRで更新する（不変性の原則）
- `docs/adr/README.md` の一覧は常に最新の状態を保つ
